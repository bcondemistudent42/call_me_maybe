# Call me maybe

## Description
`Call_me_maybe` is a Python project that converts natural-language prompts into structured function calls.

Given a list of prompts and a catalog of callable functions with typed parameters the program returns JSON objects with exactly:
- `prompt`
- `name`
- `parameters`

The goal is to make LLM output machine usable and parseable.

## Algorithm explanation
The decoding pipeline is split into two phases.

1. **Function selection (`get_function_name`)**
   - The prompt is encoded as well as the list of available function names.
   - At each decoding step, logits are computed from the model.
   - A token mask is built so that only tokens compatible with at least one candidate function name remain valid.
   - Invalid tokens are set to $-\infty$ and cannot be sampled.
   - The next token is chosen by argmax from numpy over masked logits.
   - Generation stops at end-of-sequence/newline.

2. **Parameter extraction (`get_function_args` + post-processing)**
   - Each expected parameter is extracted sequentially from the selected function context.
   - Raw text values are normalized.
   - The output type is either `number` to `float` or `integer` to `int`, otherwise string.
   - The final object is assembled as:
     - `prompt`: original prompt
     - `name`: selected function name
     - `parameters`: typed parameter dictionary


## Design decisions
- **Pydantic models for input validation**: `Function`, `Prompt`, `Parameters`, and `Returns` validate incoming JSON structure.
- **Modular split**:
  - `parser.py`: schema + file parsing,
  - `extractor.py`: LLM interaction and decoding,
  - `__main__.py`: CLI and orchestration.
- **Simple CLI defaults**: default data paths allow one-command execution.
- **Graceful error handling**: invalid JSON and validation errors are caught and reported.

## Performance analysis
- **Accuracy**: function-name reliability is improved by token-level masking over known candidates.
- **Speed**: generation remains lightweight.
- **Reliability**: Pydantic validation and typed parameter conversion reduce malformed outputs.

Current practical limits:
- parameter extraction is less constrained than function-name generation,
- ambiguous prompts may still produce imperfect argument values,
- could improve the constraint decoding for arugments

## Challenges faced
- **Token-level constraints**: mapping partial generation to valid next tokens required careful prefix filtering.
- **Type normalization**: converting model text into schema-compatible values needed explicit casting and fallbacks.
- **Error resilience**: handling malformed/missing JSON files without crashing.

## Testing strategy
- **Schema validation tests**: confirm that malformed function definitions and prompts are rejected.
- **Type checks**: verify output parameter types match function definitions.
- **Edge cases**:
  - empty/invalid input files,
  - missing files,
  - unusual strings and regex-like text,
  - numeric conversion failures.
- **Static checks**: run `flake8` and `mypy` through the Makefile targets.
